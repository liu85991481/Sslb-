# SSLB 项目说明

## 项目概述

SSLB 是一个以中国古代“三省六部”组织模型为灵感构建的多智能体协作系统，用来把一个复杂任务拆解为多角色、多阶段的 AI 协同流程。

项目目前包含两条实现思路：

1. **工作流编排版**：在 `my_agent_project/main.py` 中，使用 LangGraph 将多个角色节点串成一个完整流程。
2. **文件总线调度版**：在 `dispatcher.py`、各部门目录以及 `monitor.py` 中，使用 `inbox/outbox` 目录和批处理脚本驱动多个独立智能体实例协作。

核心目标是把“接收任务 → 审议方案 → 拆解执行 → 专业会办 → 最终裁决”这一套流程结构化、角色化，并让不同模型承担不同职责。

---

## 设计理念

项目将传统行政分工映射为 AI 智能体分工：

- **中书省**：起草任务方案
- **门下省**：审议、驳正、找漏洞
- **尚书省**：拆解执行、统筹分发
- **六部**：从不同专业视角并行评审或执行
  - 吏部：分工与协同
  - 户部：资源与预算
  - 礼部：表达与呈现
  - 兵部：推进与策略
  - 刑部：风险与合规
  - 工部：技术实现
- **御史台 / 终审层**：汇总会办意见并做最终裁决

这种设计适合用于：

- 复杂任务拆解
- 多维度方案评审
- 需求/项目方案生成
- 多模型协作实验
- 面向流程治理的 Agent 编排验证

## 两种实现方式

### 1. LangGraph 工作流版

`my_agent_project/main.py` 定义了一个完整的状态机工作流。

### 流程顺序

1. **中书省**：根据用户任务生成草案
2. **门下省**：对草案进行审议与批评
3. **尚书省**：把通过审议的内容拆解为执行分案
4. **六部并行**：分别从协同、资源、表达、推进、风险、技术角度输出意见
5. **御史台**：汇总六部意见并给出裁决

### 状态管理

通过 `AgentState` 保存以下关键信息：

- 原始任务 `task`
- 中书省草案
- 门下省审议结果
- 尚书省执行分案
- 六部会办意见
- 御史台裁决结果
- 当前阶段 `current_phase`
- 消息记录 `messages`

### 模型分工

根据 `my_agent_project/main.py` 中的定义：

- **GPT**：仅用于门下省逻辑审议
- **MiniMax**：用于吏部
- **Qwen**：作为主力模型，负责中书省、尚书省和多数六部节点
- **Kimi**：用于最终裁决节点，承担长上下文综合判断

这意味着项目不仅是在做多角色协作，也是在做**多模型协同分工实验**。

---

### 2. 文件总线调度版

`my_agent_project/dispatcher.py` 提供了另一种更加贴近“独立 Agent 进程协作”的方案。

### 核心机制

- 每个智能体拥有自己的 `inbox/` 与 `outbox/`
- 调度器持续轮询各目录
- 当发现某智能体 `outbox` 有结果时：
  - 写入总线日志
  - 按层级关系上报给上级智能体
  - 必要时启动对应智能体的批处理脚本
- `.bus/` 目录用于统一归档和日志管理

### 主要能力

- `write_inbox()`：向目标智能体写入任务 JSON
- `start_agent()`：启动对应 `.bat` 脚本
- `process_outbox()`：处理结果、记录日志、向上汇报
- `dispatch_command()`：主动给某个智能体派发任务
- `run_orchestrator()`：持续运行调度器

### 适用场景

这种方式更适合：

- 多个独立 Claude/Qwen 会话并行运行
- 用文件系统模拟消息队列
- 做可观察、可调试的多智能体实验
- 分层组织结构的任务流转验证

## Prompt 体系

`my_agent_project/prompts.py` 定义了全部角色提示词。

每个角色都包含：

- 角色职责
- 关注重点
- 禁止事项
- 输出结构
- JSON 输出格式要求

这说明项目并不是简单地“调用多个模型”，而是通过**强约束 Prompt** 来保证角色边界、输出一致性和结构化结果。

例如：

- 门下省被要求“必须提出真实批评，不得无条件通过”
- 尚书省强调“执行拆解与协同，不参与方案决策”
- 六部各自只能站在自己的专业视角发言

这类设计有利于减少角色串位和重复劳动。

## 运行方式

## 1. Claude Code 项目级模型配置

`core/claude-model-config.md` 说明了如何在项目目录下使用 `.claude/settings.local.json` 为 Claude Code 单独配置：

- API Key
- Base URL
- 模型名称
- 环境变量覆盖

`core/start.bat` 内容很简单：

```bat
@echo off
cd /d "%~dp0"
claude
```

作用是在指定项目目录启动 Claude，并应用该目录下的本地配置。

---

## 2. LangGraph 版运行

理论上主入口是：

```bash
python D:/sslb/my_agent_project/main.py
```

默认示例任务为：

- `开发一个用户登录系统`

运行后会依次打印：

- 三省决议
- 六部会办意见
- 御史台裁决
- 最终流程完成提示

---

## 3. 调度器模式运行

启动调度器：

```bash
python D:/sslb/my_agent_project/dispatcher.py --mode orchestrator
```

主动派发任务：

```bash
python D:/sslb/my_agent_project/dispatcher.py --mode send --from-agent human --to-agent zhongshusheng --command "执行测试任务"
```

监控状态：

```bash
python D:/sslb/monitor.py
```

## 测试脚本

项目提供了多种测试方式：

### `test_full.py`
用于对全部部门进行统一测试。

特点：

- 遍历所有智能体 Prompt
- 向对应目录写入测试任务
- 调用外部模型接口获取结果
- 提取 JSON 并写入 `outbox`

### `test_chain.py`
用于验证一条简化链路：

- 中书省 → 尚书省 → 工部

适合快速验证链式传递是否正常。

### `extract_result.py`
用于把原始模型返回文本中的 JSON 结构提取出来，转为标准结果文件，便于后续调度器消费。

### `monitor.py`
用于实时查看：

- 智能体状态
- 当前任务 ID
- inbox 数量
- outbox 数量

## 技术栈

从现有代码可以看出，项目主要使用：

- **Python 3**
- **LangGraph**：构建多节点工作流
- **OpenAI 兼容 SDK**：统一访问多家模型接口
- **python-dotenv**：加载环境变量
- **requests**：直接调用部分 HTTP API
- **Windows 批处理脚本**：拉起各智能体实例
- **文件系统消息总线**：实现轻量级任务分发

这是一个偏工程实验型的多智能体编排项目，而不是单纯的 Prompt 示例集。

## 项目特点

### 1. 组织隐喻明确
项目使用“三省六部”作为天然分工模型，结构直观，便于把复杂任务映射到职责明确的 AI 节点。

### 2. 多模型协同
不是一个模型做所有事，而是根据角色特点分配不同模型，提高任务适配度。

### 3. 结构化输出
各节点都要求给出 JSON，便于机器继续消费、路由和归档。

### 4. 同时支持流程图式与分布式式协作
既能在单进程 LangGraph 中跑通，也能模拟多会话、多进程的真实协作。

### 5. 可观测性较强
通过 `.status`、`inbox/outbox`、`.bus/logs`、`monitor.py`，可以比较直观地观察任务流转情况。

## 当前更像什么

结合目录内容判断，这个项目当前更像：

- 一个**多智能体协作框架原型**
- 一个**多模型职责分工实验场**

它已经具备角色、流程、调度、监控、测试这几类核心元素，但仍带有明显原型性质，例如：

- 项目结构分为多个实验入口
- 配置和运行方式还偏手工
- 测试脚本更多是验证链路，而不是完整自动化测试体系

## 总结

SSLB 是一个围绕“三省六部”组织模型设计的多智能体系统原型。它一方面通过 LangGraph 在单流程内完成多角色协作，另一方面通过 `inbox/outbox + dispatcher` 的方式模拟分布式 Agent 协作。项目强调：

- 角色边界
- 结构化输出
- 多模型分工
- 分层调度
- 可观察的任务流转

如果你的目标是研究“如何让多个 AI 节点像组织系统一样协同工作”，这个项目已经提供了一个很具体、很有辨识度的起点。
